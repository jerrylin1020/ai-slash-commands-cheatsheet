Claude Code 詳解

這一區都來自 Claude Code 官方 commands / interactive mode 文件。

Claude Code 官方確認

/compact

用在對話變長、context 快滿、但你不想重新開新 session 的時候。

  • 效果是把前面內容濃縮成摘要，繼續同一條工作線。
  • 跟 /clear 不同，它不是重開，而是壓縮。
例子 
/compact 請保留目前 bug root cause、已改檔案、下一步驗證項目
Claude Code 官方確認

/plan

適合先分析、先拆解，而不是立刻修改程式的情境。

  • 像是大型重構、陌生 repo、需求不清楚時特別好用。
  • 你也可以直接在指令後面附任務描述。
例子 
/plan 幫我規劃把 monolith auth 流程拆成 service + middleware
Claude Code 官方確認

/model 與 /effort

這兩個最常一起用。前者是換模型，後者是調整同一模型的思考深度。

  • /model 解決「要用哪個模型」。
  • /effort 解決「要想多深」。
例子 
/model sonnet
/effort high
Claude Code 官方確認

/permissions

當你覺得 Claude 一直問權限，或你想更保守地限制工具時，就用這個。

  • 可管理 allow / ask / deny 規則。
  • 也可以看最近 auto mode 被拒絕了哪些操作。
例子 
/permissions
Claude Code 官方確認

/resume、/rename、/clear

這三個是 session 管理組合。

  • /rename 幫當前工作命名，之後較好恢復。
  • /resume 回到舊工作脈絡。
  • /clear 則是重新開新上下文。
例子 
/rename dashboard-stage-2
/clear
/resume dashboard-stage-2
Claude Code 官方確認

/diff、/review、/usage

這三個很適合做收尾。

  • /diff 看改了什麼。
  • /review 請它幫你做 code review。
  • /usage 看花了多少用量與成本。
例子 
/diff
/review
/usage

Codex 常用入口

模型與推理深度

Codex 跟 Claude Code 一個很大的差異，是很多高頻控制不一定用 slash command 暴露，而會放在 UI 或 session controls。

  • 先看目前跑的是哪個模型，再決定要不要提高 reasoning / effort。
  • 這類控制點在實務上跟 slash commands 一樣重要，甚至更常用。
  • 所以如果你只盯著 slash commands 數量，很容易低估 Codex 的實際可控性。
使用心法 
先確認模型
再確認 reasoning
最後才決定要不要切 mode
Codex 常用入口

批准、sandbox 與自動化程度

另一個很常用但不一定長成 slash command 的控制點，是這回合可以做多深、多自動、能不能連網、能不能改檔。

  • 很多時候你真正調的不是「指令」，而是這次 session 的權限邊界。
  • 這也是 Codex 使用體驗常比單純指令表更關鍵的地方。

工具、連接器與外部能力

在 Codex 裡，很多你以為會是一條 command 的功能，實際上是靠工具或 connector 提供的。

  • 例如 GitHub、browser、calendar、drive、notion 這些能力，常常比多一條 slash command 更有價值。
  • 所以查 Codex 時，除了看 /，也要看這個環境到底開了哪些工具。
  • 這能解釋為什麼「命令少」不一定等於「能做的事少」。

  • Codex 目前官方最清楚提到的是 /mode，但其他高頻控制常在 UI、設定與批准邊界裡。
